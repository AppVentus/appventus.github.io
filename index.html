Introduction

This website collect numbers of rules and examples about the code should be written, architectured and thought for the AppVentus front-end team & each AppVentus project.

For each statement the idea is to keep a code coherent in order to make sure everyone's efficency. Our goal is to keep clear and easy to maintain. Modulable and Scalable should be the philosophy in order to avoid structural problems and makes components movable to any other project.

Of course, any rule of this document can be discussed, feel free to improve any following statements.

html

Synthax

  • Always use double quotes, never single quotes, on attributes.
  • Don't include a trailing slash in self-closing elements—the HTML5 spec says they're optional. 
<ul>
    <li>
        <img src="image.png">
    </li>

    <li>
        <a href="#">see more</a>
    </li>
</ul>

Indentation

  • Use 4 spaces for indentation — they're the only way to guarantee code renders the same in any environment.
  • Never indent inline a native display: block; element (<div>, <p>). Only write inline the natives display: inline; or display: inline-block; in order to clarify html semantics. 
<p>
····lorem ipsum<span>dolor</span>
</p>

Using Bootstrap grid

  • Any .col-*-* must have .row as direct parent.
  • Avoid usage of any other class or id on .row and .col-*-* elements. 
<!-- Not so good -->
<div class="pod">
    <div class="col-xs-8 pod-content">
        lorem
    </div>

    <div class="col-xs-4 pod-sidebar">
        ipsum
    </div>
</div>

<!-- Much better -->
<div class="pod">
    <div class="row">
        <div class="col-xs-8">
            <div class="pod-content">
                lorem
            </div>
        </div>

        <div class="col-xs-8">
            <div class="pod-sidebar">
                ipsum
            </div>
        </div>
    </div>
</div>

twig

Synthax

  • Space between after entering delimiters ({{ or {%) and before ending delimiters (}} or %}).
  • Reminds into the {% endblock %} tag the block's name.
  • For filters usage, don’t use spaces after and before the filter marker (|)
  • Use space between the concatenate operator (~) 
<div class="user">
    {{ user.firstname|trim ~ ' ' ~ user.lastname|trim }}
</div>

Indentation

  • Indent twig tags as any markup element (without carring about de processed markup indentation).
  • Never indent twig tags like inline html elements.
  • Blocks into html tag as attributes can be inline 
<ul>
    {% for item in items %}
        <li{% if loop.first %} class="first"{% endif %}>
            {{ item.title }}
        </li>
    {% endfor%}
</ul>

Prefer twig comments instead of html comment in order to not print out into the browser. 

{# Profile content #}
<div id="profile">
    ...
</div>

Block declaration

  • Reminds into the endblock tag the name of block defined
  • Start & end block should be both on the same line when it's markup free 
{{ block content }}{{ endblock content }}

For includes, in case of multiple variables commited with the include indent it on serveral lines likes arrays. 

{% include'_nav.html.twig' with {
    'summary': summary,
} %}
  • Avoid javascript into twig templates.
  • Avoid "static string”, rather use translate keys.
  • Use asset() function for images includes.
  • Use assetic for stylesheet includes. 
<img src="{{ asset('logo.png') }}">

File Naming

  • Use camelCase, not underscores, for files naming.
  • Prefix with "_" partials files as any template that can not be rendered stand-alone. 
{% include '_header.html.twig' %}

Use camelCase, not underscores, for variable naming. 

{% set userFullName = user.firstName ~ ' ' ~ user.lastName %}

<div class="user">
    {{ userFullName }}
</div>

css

Synthax

  • Place closing braces of declaration blocks on a new line.
  • Include one space after : for each declaration.
  • Each declaration should appear on its own line for more accurate error reporting.
  • End all declarations with a semi-colon. The last declaration's is optional, but your code is more error prone without it.
  • Comma-separated property values should include a space after each comma (e.g., box-shadow).
  • Don't prefix property values or color parameters with a leading zero (e.g., .5 instead of 0.5 and -.5px instead of -0.5px).
  • Quote attribute values in selectors, e.g., input[type="text"].
  • Never do inline declarations. 
input {
    [type="text"] {
        opacity: .8;
        border-width: 1px;
        border-style: solid;
        border-color: $gray;
        @include box-shadow (0 0 2px 3px $anthracite);
    }
}

Indentation

  • Use 4 spaces for indentation.
  • When grouping selectors, keep individual selectors to a single line.
  • Include one space before the opening brace of declaration blocks for legibility. 
h1,
.h1 {
····margin-bottom:·0;
····font-size:·4em;
····text-shadow:·none;
}

Strive to limit use of shorthand declarations to instances where you must explicitly set all the available values. Common overused shorthand properties include:

  • padding
  • margin
  • font
  • background
  • border
  • border-radius
/* Not that good */
.container {
    margin: 10px auto 0;
}

/* Much better */
.container {
    margin-top: 10px;
    margin-left: auto;
    margin-right: auto;
}

Responsive Usages

  • Place media queries as close to their relevant rule sets whenever possible.
  • In media queries specify media type (like all or screen) only if necessary 
.nav > li + li {
    margin-left: 20px;
}

@media (max-width: 768px) {
    .nav > li + li {
        margin-left: 10px;
    }
}

Using colors

  • use hexa or rgba formats for colors definition.
  • Use shorthand hex values where available (e.g., #fff instead of #ffffff.) 
fieldset {
    background-color: rgba(0, 0, 0, .5);
}
  • Use id’s for layout purposes only
  • Avoid specifying units for zero values (e.g., margin-top: 0; instead of margin-top: 0px;). 
/* Not that good */
ul {
    margin-bottom: 0px;
}

/* Much better */
ul {
    margin-bottom: 0;
}

scss

Indentation

Use sass and less nesting for every selectors in order to make your selector deepness clear and loud. If you indent more than three time for a nested selected, your selector deepness should be to strong and your code design must be wrong. 

.h1 {
    .small {
        margin-top: 15px;
    }
}

Operations

For improved readability, wrap all math operations in parentheses with a single space between values, variables, and operators. 

.pod {
    margin-top: (10px + 5px);
}

Use variables for colors references. 

$black: #000;
$black-fade: transparentize($black, .5);

fieldset {
    background-color: $black-fade;
}

Parameters Variables naming conventions (using camelCase): $element-subelement_property-_variant. 

$h1-small_fontSize: .85em;

.h1 {
    .small {
        font-size: $h1-small_fontSize;
    }
}

Do not manually prefix attributes, use third party mixins as in less-hat or bourbon.scss or task runner like gulp or grunt. 

input {
    [type="text"] {
        @include box-shadow (0 0 2px 3px $anthracite);
    }
}

Responsive usages

Use media queries inside selector and don’t add any other selector inside the media queries, only property and values 

.nav {
    > li {
        + li {
            margin-left: 20px;

            @media (max-width: 768px) {
                margin-left: 10px;
            }
        }
    }
}
